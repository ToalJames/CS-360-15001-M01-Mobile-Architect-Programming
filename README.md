# CS-360-15001-M01-Mobile-Architect-Programming
CS-360-15001-M01 Mobile Architect &amp; Programming Final Project

App Requirement and Goals

The Warehouse Inventory Tracker is designed to meet the needs of logistics for small and large companies that require a streamlined way to manage stock. The primary goal was to keep the design as simple as possible with no unnecessary add-ons. I feel that this was accomplished well, however, I feel it is extremely unfinished and could be so much better.

User-Centered UI Design

The UI design and choices are built around ensuring easy visibility and interaction, reflecting the reality that users will be in a business warehouse attempting to accomplish their production goals. To assure this, we maintain a color scheme that matched, large keys for easy clicking, and kept the navigation as quick and easy as possible.

Coding approach and strategies

My development process followed and focused on modularity; I believe this is the correct approach at all times. Break down the larger project into small chunks. Unfortunately, when it comes to mobile apps, they are almost always intertwined, and that's when I start having issues with my code. I misremembered the name spacing on one when it needed to link to another, so I had to go back, find my mistake, and correct it. This is why modularity is important: if I hadn't noticed this sooner, it would have been worse later.

Testing and Validation

My functionality was mainly tested through manual unit testing and the Android Studio Logcat tool. This was critical as I had a critical failure crash at one point during creation. Once the code was functioning, I manually used it myself, sort of how people would use testers in real society to ensure the code was functioning and made some changes, because at one point I realized it wasn't doing what I intended and would not have noticed till I tested it.

Innovation and Problem-Solving

The hardest issues I ran into were the critical crashes and failures; these were the result, as previously mentioned, of incorrect namespace references in my code. This made me stumble for a while before I finally managed to locate the issue; following that, it was how to implement the SMS allowance screen. I eventually settled on making it trigger automatically upon entering the application to ensure we had permission when necessary.

Demonstrating Proficiency

While I am proud of the way I meant to handle back-end integration, overcome my critical code failures, and demonstrated my ability to handle such a massively overwhelming project, I realized during my Launch Plan section, there is so much more to be done, and so much more I would want to do, and I believe real clients would want.
